Introduction to FastAPI

What is FastAPI?

FastAPI is a modern, fast (high-performance) web framework for building APIs with Python 3.6+ based on standard Python type hints. It builds on top of Starlette (web framework) and Pydantic (data validation library).

Key Definition: FastAPI is an ASGI-based (Asynchronous Server Gateway Interface) Python framework that combines the speed of Node.js with the simplicity and expressiveness of Python.

Why FastAPI?

‚ö° Performance

Benchmarked as one of the fastest Python frameworks, comparable to NodeJS and Go. Supports async/await natively.

‚úÖ Data Validation

Automatic validation using Pydantic models with clear error messages and JSON schema generation.

üìö Auto Documentation

Automatic interactive API documentation (Swagger UI & ReDoc) at /docs and /redoc endpoints.

üîê Security

Built-in support for OAuth2, API keys, JWT tokens, and CORS with minimal setup.

üéØ Type Hints

Full type hints support for better IDE autocomplete, type checking, and code documentation.

üöÄ Async Support

Native async/await support for handling concurrent requests efficiently without blocking.

REST APIs & HTTP Methods

What is REST?

REST (Representational State Transfer) is an architectural style for designing networked applications using HTTP requests to perform CRUD (Create, Read, Update, Delete) operations on resources identified by URLs.

REST Principles:
  • Client-Server: Clear separation between client and server
  • Statelessness: Each request contains all necessary information
  • Resource-Based: URLs represent resources, not actions
  • Standard Methods: HTTP methods define operations (GET, POST, PUT, DELETE)
  • Representation: Resources represented in multiple formats (JSON, XML)

HTTP Methods Overview

Method Purpose Body Idempotent Safe Use Case
GET Retrieve resource No Yes Yes Fetch data
POST Create resource Yes No No Create new item
PUT Update resource Yes Yes No Replace entire resource
PATCH Partial update Yes No No Partial modifications
DELETE Remove resource Optional Yes No Delete item

3xx Redirection

  • 301: Moved Permanently
  • 302: Found - Temporary redirect

5xx Server Error

GET Requests: Retrieving Data

What is GET?

GET is an HTTP method used to retrieve data from a server. It's a safe operation that doesn't modify server state and should be idempotent (multiple calls return same result).

GET Characteristics

Safe: Doesn't modify data
Idempotent: Multiple calls = same result
Cacheable: Can be cached by browsers
No Body: Data in URL only
Bookmarkable: Can be shared as URL
Limited Data: URL length limitations

POST Requests: Creating Data

What is POST?

POST is an HTTP method used to submit data to the server to create a new resource. Unlike GET, POST requests include a request body and are not idempotent (multiple calls can create multiple resources).

POST Characteristics

Not Safe: Creates/modifies data
Not Idempotent: Multiple calls create multiple resources
Has Body: Data in request body
Not Cacheable: Usually not cached
Status 201: Returns Created status
Data Validation: Server validates request

Basic POST Example

Python - Basic POST
from pydantic import BaseModel from fastapi import FastAPI, HTTPException from typing import Optional app = FastAPI() class UserCreate(BaseModel): name: str email: str age: Optional[int] = None # In-memory database users_db = {} next_user_id = 1 @app.post("/users", status_code=201) async def create_user(user: UserCreate): """ Create a new user Request Body: { "name": "Charlie", "email": "charlie@example.com", "age": 30 } Response: {"id": 3, "name": "Charlie", ...} """ global next_user_id user_id = next_user_id next_user_id += 1 users_db[user_id] = { "id": user_id, "name": user.name, "email": user.email, "age": user.age } return users_db[user_id]

PUT Requests: Updating Data

What is PUT?

PUT is an HTTP method used to replace an entire existing resource with new data. It's idempotent, meaning multiple identical PUT requests will have the same effect as a single request.

PUT vs PATCH

PUT (Replace)

  • ‚úì Replaces entire resource
  • ‚úì Idempotent operation
  • ‚úì Requires all fields
  • ‚úì Clear and predictable
  • ‚úì Standard HTTP semantics

PATCH (Partial Update)

  • ‚úì Updates specific fields
  • ‚úì Only provide changed fields
  • ‚úì More efficient for large objects
  • ‚úì Flexible updates
  • ‚úì Reduces bandwidth

PUT Implementation

Python - Complete PUT Example
from pydantic import BaseModel, Field from typing import Optional from fastapi import HTTPException class UserUpdate(BaseModel): name: str = Field(..., min_length=2, max_length=50) email: str age: int # Database users_db = { 1: { "id": 1, "name": "Alice", "email": "alice@example.com", "age": 25 } } @app.put("/users/{user_id}") async def update_user(user_id: int, user: UserUpdate): """ Replace entire user resource Requires ALL fields to be provided Returns updated user or 404 if not found Request: PUT /users/1 Body: { "name": "Alice Updated", "email": "alice.updated@example.com", "age": 26 } """ if user_id not in users_db: raise HTTPException( status_code=404, detail=f"User {user_id} not found" ) # Replace entire record users_db[user_id] = { "id": user_id, "name": user.name, "email": user.email, "age": user.age } return users_db[user_id]

PATCH for Partial Updates

Python - PATCH Implementation
from pydantic import BaseModel from typing import Optional class UserPatch(BaseModel): # All fields optional for partial updates name: Optional[str] = None email: Optional[str] = None age: Optional[int] = None @app.patch("/users/{user_id}") async def partial_update_user(user_id: int, user_update: UserPatch): """ Partially update user (only provided fields) Request: PATCH /users/1 Body: {"name": "New Name"} # Only update name Other fields remain unchanged """ if user_id not in users_db: raise HTTPException(status_code=404) # Update only provided fields user_data = users_db[user_id] update_data = user_update.dict(exclude_unset=True) for field, value in update_data.items(): if value is not None: user_data[field] = value return user_data

PUT vs POST for Creation

Aspect POST (Create) PUT (Create or Replace)
Idempotent No Yes
Multiple Calls Creates multiple resources Creates/replaces one resource
Client Controls ID No Yes
Status Code 201 Created 200 OK or 204 No Content
Use Case Creating new resources Replacing or creating with known ID

DELETE Requests: Removing Data

What is DELETE?

DELETE is an HTTP method used to remove a resource from the server. It's idempotent and typically returns a 204 No Content status code on successful deletion.

DELETE Characteristics

Idempotent: Multiple deletes = safe
Not Safe: Modifies server state
204 Status: No content returned
No Body: Response body typically empty
Resource ID: Path parameter required
Cascading: May delete related data

Basic DELETE Implementation

Python - Basic DELETE
from fastapi import HTTPException users_db = { 1: {"id": 1, "name": "Alice"}, 2: {"id": 2, "name": "Bob"} } @app.delete("/users/{user_id}", status_code=204) async def delete_user(user_id: int): """ Delete a user by ID Returns: - 204 No Content on success - 404 Not Found if user doesn't exist Usage: DELETE /users/1 """ if user_id not in users_db: raise HTTPException( status_code=404, detail=f"User {user_id} not found" ) del users_db[user_id] # Return None with status_code 204

DELETE with Cascading Deletes

Python - Cascading Deletes
posts_db = { 1: {"id": 1, "user_id": 1, "title": "Post 1"}, 2: {"id": 2, "user_id": 1, "title": "Post 2"} } @app.delete("/users/{user_id}", status_code=204) async def delete_user_cascade(user_id: int): """ Delete user and all related posts (cascading delete) """ if user_id not in users_db: raise HTTPException(status_code=404) # Delete user del users_db[user_id] # Delete all posts by this user post_ids_to_delete = [ pid for pid, post in posts_db.items() if post["user_id"] == user_id ] for post_id in post_ids_to_delete: del posts_db[post_id]

DELETE with Return Data

Python - DELETE with Response Data
from pydantic import BaseModel from typing import Dict class DeleteResponse(BaseModel): message: str deleted_id: int timestamp: str @app.delete("/users/{user_id}", response_model=DeleteResponse) async def delete_user_with_response(user_id: int): """ Delete user and return deletion confirmation Returns 200 OK with response body (instead of 204) """ if user_id not in users_db: raise HTTPException(status_code=404) user = users_db[user_id] del users_db[user_id] from datetime import datetime return DeleteResponse( message=f"User {user['name']} deleted successfully", deleted_id=user_id, timestamp=datetime.now().isoformat() )

Bulk DELETE Operations

Python - Bulk Delete
from typing import List @app.delete("/users/bulk") async def bulk_delete_users(user_ids: List[int]): """ Delete multiple users Request: DELETE /users/bulk Body: {"user_ids": [1, 2, 3]} """ deleted_count = 0 not_found_ids = [] for user_id in user_ids: if user_id in users_db: del users_db[user_id] deleted_count += 1 else: not_found_ids.append(user_id) return { "deleted_count": deleted_count, "not_found": not_found_ids }

Safe DELETE with Soft Delete

Python - Soft Delete Pattern
from datetime import datetime # Add is_deleted and deleted_at fields users_db = { 1: { "id": 1, "name": "Alice", "is_deleted": False, "deleted_at": None } } @app.delete("/users/{user_id}", status_code=204) async def soft_delete_user(user_id: int): """ Soft delete: Mark as deleted instead of removing Advantages: - Data recovery possible - Maintains referential integrity - Audit trail preserved - Can restore deleted items """ if user_id not in users_db: raise HTTPException(status_code=404) user = users_db[user_id] user["is_deleted"] = True user["deleted_at"] = datetime.now() # When querying, exclude soft-deleted items @app.get("/users") async def list_users(): """List only non-deleted users""" return [ user for user in users_db.values() if not user.get("is_deleted", False) ]

What is a Vector Database?

A vector database is a specialized database designed to store, index, and search high-dimensional vectors (embeddings). It enables similarity-based searches, crucial for AI/ML applications, semantic search, and recommendation systems.

Vector Database Definition: A database optimized for storing and retrieving data based on similarity of numerical vectors rather than exact matches.

Key Concepts

Concept Description Example
Vector/Embedding Numerical representation of data (text, image, etc.) [0.25, -0.89, 0.12, ..., 0.55]
Dimension Number of values in a vector OpenAI embeddings: 1536D
Similarity Search Find vectors closest to query vector Semantic search results
Distance Metric Method to calculate vector similarity Cosine, Euclidean, Manhattan
Index Data structure for fast vector search HNSW, IVF, LSH

Complete CRUD Operations Guide

What is CRUD?

CRUD represents the four basic operations for managing data: Create, Read, Update, Delete. These map directly to HTTP methods: POST, GET, PUT/PATCH, DELETE.

CRUD Operations Mapping:
  • Create (C): POST - Add new resource
  • Read (R): GET - Retrieve existing resource
  • Update (U): PUT/PATCH - Modify existing resource
  • Delete (D): DELETE - Remove resource
